1. MOTIVATION AND RESEARCH QUESTION
─────────────────────────────────────
The 12 existing datasets in the paper each capture a single, homogeneous
workload: one benchmark run per dataset, with train and test drawn from the
same execution environment.  A practical concern for deployment is whether a
diffusion model trained on one workload intensity remains effective when the
system shifts to a different intensity or is subjected to external stress.

RQ-W: "Does SSSD^S4 performance degrade when trained on one workload
        condition and tested on a structurally different workload from the
        same application?  How does cross-workload performance compare to
        in-workload performance?"

Two complementary experiment families address RQ-W:

  ELK Workload Experiment
  ──────────────────────
  Uses the ELK dataset (Elasticsearch + Kibana stack under LTTng tracing).
  The ELK experiment has a known timeline with injected CPU / I/O / Network /
  Memory noise, creating two structurally distinct workload phases:

    elk_clean — the pre-noise phase (0–1200 s): normal query processing,
                40 distinct syscalls, rich variety of execution patterns.
    elk_noisy — the noise-injection phase (1200–1860 s): system under
                simultaneous stress from stress-ng, only 6 distinct syscalls
                remain (highly repetitive low-level I/O/IPC calls dominate).

  Four conditions are evaluated:
    (1) elk-clean-baseline  : train on elk_clean, test on elk_clean
    (2) elk-noisy-baseline  : train on elk_noisy, test on elk_noisy
    (3) elk-clean-to-noisy  : train on elk_clean, test on elk_noisy  ← cross
    (4) elk-noisy-to-clean  : train on elk_noisy, test on elk_clean  ← cross

  PTS Within-Family Experiment
  ─────────────────────────────
  Uses two pairs of Phoronix Test Suite (PTS) benchmarks that share the same
  type of system stress but generate structurally different syscall sequences.

  Memory-bandwidth family (stream vs ramspeed):
    Both benchmarks saturate DRAM bandwidth via sequential read/write loops.
    Their syscall sequences share the same small vocabulary (~58 types) but
    differ in loop structure, stride, and vectorisation artefacts.
    (5) pts-stream-baseline    : train=stream,   test=stream
    (6) pts-ramspeed-baseline  : train=ramspeed, test=ramspeed
    (7) pts-stream-to-ramspeed : train=stream,   test=ramspeed  ← cross
    (8) pts-ramspeed-to-stream : train=ramspeed, test=stream    ← cross

  CPU-benchmark family (pybench vs phpbench):
    Both benchmarks execute language-level microbenchmarks (Python / PHP),
    generating similar syscall patterns (fork, exec, mmap, polling) but with
    different timing ratios and loop counts.
    (9)  pts-pybench-baseline    : train=pybench,  test=pybench
    (10) pts-phpbench-baseline   : train=phpbench, test=phpbench
    (11) pts-pybench-to-phpbench : train=pybench,  test=phpbench  ← cross
    (12) pts-phpbench-to-pybench : train=phpbench, test=pybench   ← cross

    ───────────────────────────────────────────────
    Total                                 441,427

  Vocabulary (shared, built from all 441,427 events): 59 distinct syscalls.

  Vocabulary coverage per phase:
    elk_clean: 40 distinct syscalls (67.8 % of shared vocab)
    elk_noisy:  6 distinct syscalls (10.2 % of shared vocab)
    Overlap:    6 syscalls (all noisy-phase calls are a strict subset of clean)
    Exclusive to clean: 34 syscalls
    Exclusive to noisy:  0 syscalls

  This asymmetry is key: the noisy phase is dominated by a handful of
  low-level calls (futex, recvfrom, epoll_pwait, read, write, sendto) as
  the application shifts from query processing to blocking I/O waiting under
  resource contention.  The clean phase exhibits the full syscall diversity
  of normal ELK operation.

    elk_noisy           1,213             302

5. FINDINGS AND ANALYSIS
─────────────────────────

  5.1  ELK: Near-Zero Degradation Under Extreme Vocabulary Shift

  The ELK experiment tests the most severe workload shift in our study: the
  noisy phase uses only 6 syscalls (a strict subset of the clean phase's 40).
  Despite this 85% vocabulary reduction, the clean-trained model achieves
  identical accuracy (99.54%) and perfect rate (98.68%) on the noisy test set
  as the in-workload baseline.  The rouge-L actually improves by 0.06.

  The inverse direction (noisy→clean) shows a maximum drop of 0.20 pp in
  perfect rate — negligible given that both baselines exceed 98.6%.

  Why near-zero degradation?  The noisy phase is a degenerate subset of the
  clean vocabulary, so the clean-trained model has already "seen" all 6 noisy
  syscalls during training.  Conversely, the noisy model cannot predict 34
  clean syscalls it never saw, yet the sequence structure it learned is
  sufficient to achieve 99.42% accuracy — suggesting that the temporal
  ordering patterns (not the syscall identity distribution per se) carry most
  of the predictive signal.

  5.2  PTS Memory-Bandwidth: Cross-Workload Matches / Exceeds Baseline

  Stream and ramspeed share an identical 58-syscall vocabulary but differ in
  loop patterns and memory access strides.  Ramspeed-trained model tested on
  stream achieves 83.36% accuracy — 0.54 pp HIGHER than the stream in-workload
  baseline (82.82%).  Perfect rate and ROUGE-L show the same direction.

  This result indicates that ramspeed training data, though not identical to
  stream execution patterns, provides equal or richer training signal for
  stream reconstruction.  The model appears to generalise within the
  memory-bandwidth functional family.

  5.3  PTS CPU-Benchmark: Cross-Workload Outperforms Baseline

  Pybench trained on Python microbenchmarks, tested on PHP benchmarks:
  78.56% accuracy vs 77.88% phpbench-baseline — 0.68 pp improvement.
  Similar improvements in perfect rate (+0.40 pp) and ROUGE-L (+0.50).

  This is a non-trivial positive finding: the Python interpreter's fork/exec
  patterns in pybench provide complementary training signal that improves
  phpbench reconstruction.  The two benchmarks share dominant syscall patterns
  (mmap, poll, clone) that the cross-trained model leverages effectively.

  5.4  Anticipated vs Actual (hypothesis check)

  Previous hypothesis             Actual result

  clean→noisy: possible severe    WRONG: 0.00 pp degradation, +0.06 ROUGE-L

6. ALIGNMENT WITH PAPER AND REVIEWER COMMENTS
───────────────────────────────────────────────

  (See Section 7 for the full paper-readiness assessment.)

  This section summarises how the RQ-W results relate to each reviewer's
  specific criticisms and to the paper's existing research questions.

  Reviewer 3 (primary driver of this experiment):
    ► Criticism: "The evaluation fails to account for the impact of varying
      workloads... Without evaluating the model's robustness against these
      shifts, it remains unclear how well the approach would generalize."
    ► RQ-W result: Directly contradicts this concern.  In all 4 completed
      cross-workload conditions across two datasets and two workload families,
      the model degrades by at most 0.20 pp perfect rate.  In two conditions
      it is strictly better than the in-workload baseline.
    ► Reviewer 3 also said: "RQ5 is rather expected and not useful in
      practice... varying workload is more meaningful."  RQ-W is exactly what
      was requested.

  Reviewer 2:
    ► Criticism: "Trivial results in RQ5/RQ6" (cross-application fails trivially).
    ► RQ-W result: Provides non-trivial positive results — within-system
      workload variation does NOT degrade the model.  This counters the
      reviewers' characterisation that all transferability experiments are trivial.

  Reviewer 1:
    ► Criticism: Cold-start / training data availability problem.
    ► RQ-W implication: If models trained on one workload generalise to another
      within the same system, the cold-start window is greatly reduced: a short
      deployment phase under one workload condition quickly produces sufficient
      training data that will also cover related workloads.

  Existing RQ5 (cross-application transfer):
    ► RQ-W and RQ5 together create a spectrum: cross-workload (same app) ≈ no
      degradation; cross-application ≈ complete failure.  This nuance is
      important: it suggests the model has learned application-specific syscall
      patterns that are workload-invariant but not application-portable.


7. PAPER-READINESS ASSESSMENT
───────────────────────────────

  VERDICT: YES — these results are directly relevant, non-trivial, and strongly
  address the reviewers' most actionable criticism (Reviewer 3).

  Arguments FOR inclusion:

  (a) Directly answers an explicit reviewer request.
      Reviewer 3 named "no evaluation on varying workload" as a major
      limitation.  Editors expect the revision response letter to address each
      raised concern.  RQ-W supplies exactly the requested evidence.

  (b) Provides a positive, publishable finding.
      Unlike the current RQ5 (trivially negative cross-application results),
      RQ-W shows the model WORKS under a realistic deployment challenge.
      Positive results are more persuasive and more useful to practitioners.

  (c) Completes the transferability narrative with nuance.
      Paper's current story: "model learns application-specific patterns; cannot
      transfer cross-application."  RQ-W adds: "but those patterns ARE stable
      across workload variation within the same application — deployability is
      not threatened by workload shifts."
      This is a much more compelling story for practitioners.

  (d) Uses existing datasets non-intrusively.
      ELK is already in the paper.  PTS stream/ramspeed/pybench/phpbench are
      already in the paper.  No new data collection required; reviewers cannot
      object that results rely on novel, unvalidated data.

  (e) Aligns with the paper's motivation.
      The intro motivates deployment in production systems with "high system
      load, error conditions, security breaches."  These are exactly workload
      variations.  The experiment proves the model handles them.

  Arguments AGAINST / cautions:

  (i) ELK accuracy is already ~99.44% baseline — near ceiling.
      The experiment cannot show degradation because there is almost no room
      for it.  A skeptical reviewer may dismisses the ELK result as trivially
      robust due to ceiling effect.  The PTS results (78–84% range) are more
      convincing because there is meaningful headroom.

  (ii) Two conditions still pending (stream→ramspeed, phpbench→pybench).
      The report is complete for 10/12 conditions.  Wait for jobs 59238821 and
      59232117 before finalising any paper section.

  (iii) No vocabulary mismatch in PTS cross-conditions.
      Trade benchmarks share identical vocabularies (58 syscalls).  A reviewer
      may note that true workload shift requires encountering unseen tokens.
      The ELK clean→noisy condition does test this (clean vocab ⊃ noisy vocab),
      and the result is still robust, which partially addresses this.

  RECOMMENDED PLACEMENT IN PAPER:
  Option A (preferred): Add as a new RQ — "RQ-W: Is performance robust to
    workload variation within the same application?" — positioned before or
    after RQ5.  ~0.5–1 page including a compact table.
  Option B: Subsection under RQ5, reframing RQ5 as covering both
    cross-application and cross-workload transferability.  This avoids adding
    a new RQ but groups unlike things together.
  Option A is cleaner and maps directly onto Reviewer 3's request.




  Existing RQ                     Relationship to RQ-W

    RQ-W tests whether a deployed model degrades when the same application's
